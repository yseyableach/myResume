SRBG / S360 ETL Job Management

排程平台化與參與範圍
← 回到履歷

本頁整理 SRBG / S360 排程管理從 VM Cronjob 模式轉向 ETL Job Management Console、ADF 與 Pod 架構的背景、設計方向與參與範圍。內容重點放在排程可視化、相依性管理、失敗重啟、成本分析，以及舊系統版本升級與工具轉換。

背景

早期 S360 / SRBG 的排程多以 VM 為執行單位。當排程時間到達或任務被觸發後，VM 會啟動並執行 VM 內部的 Cronjob 程式；程式初始化後會到 MDB 查詢自身需要執行的排程設定，接著執行各專案的業務邏輯，最後將結果寫回各自的應用資料庫或記錄系統。

這種模式可以支援既有業務運作，但排程狀態、相依關係、失敗原因與重啟流程分散在 VM、資料表、Log 與不同專案中，較難從單一入口掌握整體資料流。

主要問題

排程不可視化

使用者需要透過 VM、DB、Log 或專案文件確認排程狀態，較難快速掌握單一排程或整體資料流的執行情況。

相依性難以追蹤

各專案排程彼此相依，若前置節點失敗，後續排程可能無法執行，但缺乏集中化工具協助定位失敗位置與影響範圍。

VM 負載與舊系統限制

既有排程長期運行於 VM 與舊版 Node.js 環境，且每日皆有任務執行，版本升級與工具轉換需要避免影響既有排程。

管理與維運成本

排程設定、重啟、成本與事件追蹤分散，增加 RD 與使用者之間確認問題、重跑任務與驗證結果的成本。

To-Be 設計方向

To-Be 方向是將排程管理從分散在 VM 與各專案中的操作，整理成 ETL Job Management Console。透過平台化入口管理 batch 設定、排程狀態、相依性、重啟流程與成本資訊，讓 RD、PM 與業務團隊可以更快理解排程執行狀態與問題位置。

在執行架構上，逐步導入 ADF、Pod 與 Azure Function 等執行方式，分擔既有 VM 的壓力。對於可轉換的任務，將其遷移到較容易管理與觀測的執行模式；對於仍需留在 VM 的任務，則透過平台統一管理設定與狀態。

參與範圍

Batch Management

參與 Batch setting master/detail API、排程新增與編輯、DTO 檢核、排程設定維護與 Management Console 相關功能。

Dependency & Restart

參與 parent-child dependency tree、loop check、step restart、失敗任務重啟與指定專案重啟等流程。

ADF / Pod / VM 轉換

協助團隊將部分可轉換排程導入 ADF 或 Pod 架構，並保留 VM 任務的管理與追蹤能力。

版本升級與工具轉換

面對舊系統與不中斷排程限制，協助進行版本升級、工具轉換、測試驗證與環境差異控管。

產出與影響

這個方向讓排程管理從原本偏向 VM 與 DB 操作的模式，逐步轉為可由 Management Console 管理與觀測的方式。排程設定、相依關係、失敗重啟與成本資訊被集中到同一個平台中，降低人工查詢與跨團隊確認成本。

同時，透過 ADF、Pod 與 Azure Function 分擔部分 VM 任務，讓團隊在不中斷既有排程的前提下，逐步推進舊系統升級與執行架構轉換。
